# 저장소 지침

## 현재 작업 환경

- 운영체제: Microsoft Windows NT 10.0.19045.0
- shell/terminal: PowerShell 5.1.19041.6926
- 프로젝트 루트: `C:\AI\codex_test1`
- 주요 런타임: `python`, `node`, `npm`, `git`

## 기본 작업 원칙

- 작업 범위는 현재 프로젝트 루트와 하위 파일로만 한정한다.
- 모든 설명, 계획, 보고는 기본적으로 한국어로 작성한다.
- 프로젝트 내부 파일을 직접 읽고 판단한다. 추정으로 구현하지 않는다.
- 요구사항이 모호하면 먼저 짧게 확인하고, 확인이 어려우면 안전한 최소 범위로 작업한다.
- 작업 전에는 필요한 가정을 짧게 밝히고, 다중 해석이 있으면 숨기지 않는다.
- 요청 범위 밖의 기능, 설정, 추상화는 추가하지 않는다.
- 한 번에 해결되는 단순한 작업은 단순하게 처리하고, 과한 유연성은 넣지 않는다.
- 기존 코드, 주석, 포맷은 필요 이상으로 건드리지 않는다.
- 변경으로 생긴 미사용 코드만 정리하고, 기존의 무관한 죽은 코드는 건드리지 않는다.
- 다단계 작업은 짧은 계획과 검증 기준을 먼저 정리한다.
- 코드 수정 후에는 관련 테스트 또는 최소 검증을 수행한다.
- 테스트를 실행할 수 없으면 그 이유를 명확히 기록한다.
- 변경 내용과 검증 결과를 마지막에 짧게 요약한다.
- `.env`, secret, credential, API key, 인증 파일은 출력하거나 수정하지 않는다.
- 삭제, 강제 덮어쓰기, 원격 push 같은 파괴적 작업은 수행하지 않는다.
- 사용자가 제공한 변수명, 함수명, 파일명은 임의로 바꾸지 않는다.
- 불필요한 대규모 리팩토링은 하지 않는다.

LLM이 코딩 작업에서 자주 저지르는 실수를 줄이기 위한 개발 운용 원칙이다. 
프로젝트별 지시사항이 있다면 이 문서와 병합해서 사용하라.

**운용 기준:** 이 원칙은 속도보다 신중함을 우선한다. 단순하고 사소한 작업에서는 상황에 맞게 판단하라.

## 1. 코딩하기 전에 먼저 판단하라

**가정하지 마라. 혼란을 숨기지 마라. 선택지와 트레이드오프를 드러내라.**

구현을 시작하기 전에 다음을 수행하라.

* 네가 전제한 가정을 명시하라.
* 불확실한 부분이 있으면 질문하라.
* 해석이 여러 개 가능하다면 조용히 하나를 고르지 말고 가능한 해석들을 제시하라.
* 더 단순한 접근이 있다면 먼저 말하라.
* 사용자의 요구가 과하거나 방향이 잘못되었다고 판단되면 근거를 들어 반박하라.
* 명확하지 않은 부분이 있으면 멈춰라.
* 무엇이 헷갈리는지 정확히 이름 붙여라.
* 필요한 질문을 먼저 하라.

## 2. 단순함을 우선하라

**요구사항을 해결하는 최소 코드만 작성하라. 추측성 기능을 넣지 마라.**

* 요청받지 않은 기능을 추가하지 마라.
* 한 번만 쓰이는 코드에 추상화를 만들지 마라.
* 요구되지 않은 유연성, 설정 옵션, 확장성을 넣지 마라.
* 실제로 발생할 수 없는 상황을 위한 에러 처리를 만들지 마라.
* 50줄로 해결할 수 있는 코드를 200줄로 작성했다면 다시 줄여라.

항상 다음 기준으로 점검하라.

> “시니어 엔지니어가 이 코드를 보고 과하게 복잡하다고 말할 것인가?”

대답이 “그렇다”라면 단순화하라.

## 3. 최소 변경안을 염두하라.

**반드시 필요한 부분만 수정하라. 네 변경으로 생긴 찌꺼기만 정리하라.**

기존 코드를 수정할 때 다음을 지켜라.

* 관련 없는 코드를 개선하지 마라.
* 인접 코드, 주석, 포맷을 임의로 정리하지 마라.
* 고장 나지 않은 코드를 리팩터링하지 마라.
* 기존 스타일이 마음에 들지 않아도 현재 코드베이스의 스타일을 따라라.
* 관련 없는 죽은 코드를 발견하면 삭제하지 말고 사용자에게 언급만 하라.

네 변경으로 인해 불필요해진 코드가 생기면 다음을 수행하라.

* 네 변경 때문에 unused 상태가 된 import, 변수, 함수는 제거하라.
* 기존에 이미 존재하던 죽은 코드는 요청받지 않았다면 제거하지 마라.

모든 변경 라인은 다음 질문을 통과해야 한다.

> “이 줄은 사용자의 요청과 직접 연결되는가?”

직접 연결되지 않는다면 수정하지 마라.

## 4. 목표 중심으로 실행하라

**성공 기준을 먼저 정의하고, 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 바꿔라.

* “validation 추가” → “잘못된 입력에 대한 테스트를 작성하고, 그 테스트를 통과시켜라.”
* “버그 수정” → “버그를 재현하는 테스트를 먼저 작성하고, 그 테스트를 통과시켜라.”
* “X 리팩터링” → “리팩터링 전후로 테스트가 모두 통과하는지 확인하라.”

여러 단계가 필요한 작업은 시작 전에 짧은 계획을 제시하라.

```text
1. [작업 단계] → 검증: [확인 방법]
2. [작업 단계] → 검증: [확인 방법]
3. [작업 단계] → 검증: [확인 방법]
```

강한 성공 기준은 독립적으로 작업을 반복하고 마무리할 수 있게 한다.

약한 성공 기준, 예를 들어 “작동하게 만들어라” 같은 표현은 계속 사용자 확인을 요구하게 만든다. 이런 경우에는 먼저 성공 기준을 구체화하라.

* 구현 후 실수를 고치는 대신, 구현 전에 필요한 질문이 먼저 나온다.
